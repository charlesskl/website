<objective>
Create a section choreography module that replaces ALL remaining generic `data-reveal` fade-in animations with coordinated, section-level ScrollTrigger sequences. Each section's content blocks reveal in a designed sequence — not independently.

Purpose: Generic reveal-up fades feel templated and cheap. Premium sites choreograph each section as a mini-scene: the label enters, then the heading, then body text, then supporting elements — with directional movement and varied timing. This plan replaces the IntersectionObserver-based `data-reveal` system entirely.
Output: `assets/animation/section-choreography.js` (replaces data-reveal), modifications to `assets/animations.js` (remove old IO system), CSS updates, wired into 27 HTML pages.
</objective>

The module must:
1. **CDN graceful degradation**: Check `typeof gsap !== 'undefined'` and `typeof ScrollTrigger !== 'undefined'`. If missing, return gracefully — the CSS-based reveal system in animations.js will still work as fallback.
2. **Reduced motion gate**: If `window.RR.state.hasReducedMotion`, make all reveal-up elements visible immediately (set opacity: 1, transform: none) and return.
3. **Core concept**: Instead of animating each `reveal-up` element independently, GROUP elements by their parent section. For each section, create a single `gsap.timeline()` with a `scrollTrigger` on the section, then add each child element to the timeline with calculated offsets for choreographed timing.

4. **Section detection and choreography**:

   Define a `SECTION_CONFIGS` array that maps section patterns to choreography rules:

   a) **Contact section** (contact pages — `.contact-section`):
      - Elements: `.section-label.reveal-up` → `h2.reveal-up` → `p.reveal-up` → `.contact-detail.reveal-up` (staggered)
      - Choreography: label slides up first (y:20, duration:0.5), heading slides up 0.1s later (y:30, duration:0.6), paragraph 0.1s later (y:20, duration:0.5), contact details stagger in 0.08s apart (y:25, duration:0.5)
      - Trigger: `.contact-section`, start: 'top 80%', once: true

   b) **Editorial section** (capability pages — `.editorial`):
      - Elements: `.section-label.reveal-up` → `.editorial-intro.reveal-up` → `.editorial-feat.reveal-up` (staggered)

      - Trigger: `.editorial`, start: 'top 75%', once: true
      - IMPORTANT: `.section-label.reveal-up` and `.editorial-intro.reveal-up` may also be targeted by scroll-text.js for heading reveals. Check if the element has already been processed by scroll-text.js (look for `.rr-word` or `.rr-line` child elements). If already split by SplitText, SKIP that element — scroll-text.js owns its reveal. Only animate elements that have NOT been processed by scroll-text.js.

   c) **CTA section** (capability and contact pages — `.cta-section`):
      - Elements: `.container.reveal-up` children (h2, p, .btn)
      - Choreography: heading (y:25, duration:0.6), paragraph 0.1s later (y:20, duration:0.5), button 0.15s later (y:15, duration:0.4, also opacity 0→1)
      - Trigger: `.cta-section`, start: 'top 85%', once: true
      - The `.container.reveal-up` wrapper: override its CSS opacity/transform immediately on timeline creation (gsap.set), then animate children individually

   d) **Value cards** (careers page — `.values-grid` or parent of `.value-card.reveal-up`):
      - Elements: `.value-card.reveal-up` elements
      - Choreography: cards stagger 0.1s apart, each slides up (y:40, duration:0.6) with slight scale (scale:0.97→1)
      - Trigger: parent container, start: 'top 80%', once: true

   e) **Generic fallback** — for any `.reveal-up` element NOT inside a recognized section AND not already handled by scroll-text.js:
      - Apply a directional reveal based on element position:
        - If element is in the left 40% of viewport → slide from left (x:-30)
        - If element is in the right 40% → slide from right (x:30)
        - Otherwise → slide up (y:25)
      - Duration: 0.6s, ease: 'power3.out'
      - ScrollTrigger per element, start: 'top 85%', once: true

5. **SplitText conflict avoidance**: Before adding any element to a choreography timeline, check:
   - If `el.querySelector('.rr-word, .rr-line, .rr-char')` returns anything → element is already owned by scroll-text.js → SKIP it
   - If `el.id === 'stmtAccent' || el.id === 'historyWatermark'` → owned by text-scramble.js → SKIP

6. **Override CSS reveal system**: For each element this module takes ownership of, immediately call `gsap.set(el, { opacity: 0, y: [initial offset], clearProps: 'transition' })` to override the CSS-based reveal-up styles. This prevents the CSS transition from conflicting with GSAP's control.

7. **Track all timelines and ScrollTriggers** for cleanup.
8. **kill()**: Kill all tracked timelines and ScrollTriggers. Reset all affected elements to visible (`gsap.set(el, { clearProps: 'all' })`).
9. **refresh()**: Kill all, re-init.

**Modify `assets/animations.js`** — The old IntersectionObserver-based `data-reveal` system must be neutralized so it doesn't conflict with the new GSAP-based section choreography. However, we cannot simply DELETE it because it serves as a CSS-only fallback when GSAP CDN fails to load.

Strategy: Keep the IntersectionObserver code but add a guard that skips it when GSAP + section-choreography are available.

1. At the top of the `initScrollReveal()` function, add a check:
   ```javascript
   // If GSAP choreography is available (Phase 3), let it handle reveals
   // This IO system becomes fallback-only for CDN failure scenarios
   if (typeof gsap !== 'undefined' && typeof ScrollTrigger !== 'undefined') {
     // GSAP available — section-choreography.js handles all reveals
     // Still mark remaining data-reveal elements as revealed after a delay
     // in case they're not covered by any GSAP module
     setTimeout(function() {
       targets.forEach(function(el) {
         if (!el.classList.contains('is-revealed')) {
           el.classList.add('is-revealed');
         }
       });
     }, 3000); // 3s safety net — any uncovered element becomes visible
     return;
   }
   ```

2. This ensures:
   - When GSAP loads: section-choreography.js owns all reveals via ScrollTrigger
   - When GSAP fails: The old IO system kicks in as graceful degradation (FOUND-05)
   - Safety net: After 3s, any element not yet revealed by either system gets force-revealed

**Modify `assets/shared.css`** — Keep existing `[data-reveal]` CSS rules intact (they're the CSS-only fallback). But add a rule that disables the CSS transition when GSAP is managing the element:

Add after the existing `[data-reveal].is-revealed` block:
```css
/* When GSAP choreography takes over, disable CSS transitions on reveal elements
   to prevent double-animation. GSAP sets inline styles directly. */
html.js-loaded [data-reveal].rr-choreographed {
  transition: none !important;
}
```

The section-choreography.js module should add the `rr-choreographed` class to every element it takes ownership of. This prevents the CSS transition from fighting GSAP's transform control.
