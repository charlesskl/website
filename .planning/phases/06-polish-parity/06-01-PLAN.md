# Plan 06-01: Multi-language Animation Parity Audit & Mobile Performance Optimization

## Objective

Ensure all animation modules work identically across `/en/`, `/cn/`, and `/id/` HTML variants, optimize for 60fps on desktop, enforce GPU-compositable properties only, manage `will-change` lifecycle properly (apply only during active animation, remove after), and disable complex effects (parallax, WebGL, scroll skew) on touch/coarse-pointer devices.

## must_haves

- [ ] All `/cn/` and `/id/` pages load the same animation script tags as their `/en/` counterparts (including webgl-bg.js and card-effects.js where applicable)
- [ ] No animation module contains language-specific selectors or logic — all targeting is via data attributes and class names shared across all languages
- [ ] All CSS `will-change` declarations in shared.css are either inside `:hover`/`:focus` pseudo-classes, inside `@media (pointer: fine)` blocks, or removed in favor of JavaScript-managed lifecycle
- [ ] Static `will-change: transform` in shared.css is removed from elements that are not actively animating — replaced with dynamic application in JS animation modules
- [ ] The `prefers-reduced-motion` media query in shared.css contains `* { will-change: auto !important; }` (already present — verify preserved)
- [ ] Mobile media query strips `will-change` on coarse pointer devices (already present at line 1767 — verify extended to cover all static declarations)
- [ ] All animation properties in JS modules use only `transform`, `opacity`, and `clipPath` — no `width`, `height`, `top`, `left`, `margin`, or `padding` animations
- [ ] `parallax.js`, `scroll-skew.js`, and `webgl-bg.js` exit early when `window.RR.breakpoints.isCoarsePointer` is true
- [ ] `cursor.js` and `magnetic.js` exit early when pointer is coarse or hover is none

## Tasks

<task id="1">
**Audit and fix script tag parity across all language variants**

Compare the script tags at the bottom of every `/en/` HTML page against its `/cn/` and `/id/` counterpart. Ensure:

1. Every EN page's `<script>` tags for animation modules are present in the matching CN and ID page (with correct `../` relative paths for subdirectory pages).

2. Specifically check these pages that currently may be missing `webgl-bg.js` and `card-effects.js`:
   - `cn/about.html` and `id/about.html` (compare against `about.html`)
   - `cn/careers.html` and `id/careers.html` (compare against `careers.html`)
   - `cn/contact.html` and `id/contact.html` (compare against `contact.html`)
   - All `cn/capabilities/*.html` and `id/capabilities/*.html` (compare against `capabilities/*.html`)

3. Only add script tags that the corresponding EN page actually has. The EN `about.html` does NOT have webgl-bg.js and card-effects.js — that is by design (those only run on index pages with `.cap-card` elements). Do NOT add extra scripts.

4. Check that GSAP CDN script tags (gsap.min.js, ScrollTrigger.min.js, SplitText.min.js) are present in all language variants — they already are per grep check, but verify the SplitText tag specifically.

**Verification:**
- For each EN page, diff the script tags against CN and ID counterparts
- No script tag mismatches should remain
</task>

<task id="2">
**Clean up static will-change in shared.css**

The CSS has ~15 static `will-change` declarations on elements that are not always animating. This wastes GPU memory, especially on mobile.

1. Read `assets/shared.css` and identify all lines with `will-change`.

2. Remove `will-change` from these static contexts (elements that animate intermittently, not continuously):
   - `.cap-card` and card-related selectors (handled by JS in card-effects.js)
   - `.clip-section` and editorial elements (handled by JS in scroll-reveals.js/section-choreography.js)
   - Any element that only animates on scroll trigger or hover

3. KEEP `will-change` in these cases:
   - Inside the `prefers-reduced-motion` reset block (`* { will-change: auto !important; }` at line 1370)
   - Inside the mobile strip block at line 1767-1770
   - On elements that animate continuously (e.g., scroll progress bar fill, cursor elements when pointer is fine)
   - On `:hover` or `:focus` pseudo-classes (applied only during interaction)

4. For the cursor elements (`.cursor-dot`, `.cursor-ring`), wrap their `will-change` inside `@media (pointer: fine) and (hover: hover)` so mobile devices never get cursor-related GPU promotion.

5. Move `will-change: transform` for `.page-transition-overlay` and similar clip-path transition elements into JS-managed lifecycle (the animation JS already exists in `page-transition.js`).

**Verification:**
- `grep -c 'will-change' assets/shared.css` should show significantly fewer static declarations
- No static `will-change: transform` on scroll-animated elements
- Cursor elements' will-change gated behind pointer:fine media query
</task>

<task id="3">
**Audit animation modules for GPU-compositable properties only**

Review every JS module in `assets/animation/` for animation properties. All GSAP tweens must use only GPU-compositable properties:
- ALLOWED: `transform` (x, y, scale, rotate, skew), `opacity`, `clipPath`
- FORBIDDEN during animation: `width`, `height`, `top`, `left`, `right`, `bottom`, `margin`, `padding`, `border-width`, `font-size`

Files to audit:
1. `parallax.js` — uses yPercent (transform) — OK
2. `scroll-skew.js` — uses skewY (transform) — OK
3. `scroll-reveals.js` — check for any layout-triggering props
4. `section-choreography.js` — check all stagger animations
5. `card-effects.js` — uses rotateX/Y, clipPath, opacity — OK
6. `hero.js` — check hero entrance animations
7. `text-system.js` — check SplitText animations
8. `hover-effects.js` — check hover feedback animations
9. `webgl-bg.js` — canvas element, no DOM animation — OK
10. `cursor.js` — check cursor positioning method
11. `magnetic.js` — check magnetic effect implementation

For any violations found:
- Replace `top`/`left` positioning with `x`/`y` transforms
- Replace `width`/`height` animation with `scaleX`/`scaleY`
- Replace `margin`/`padding` with transforms or clip-path

**Verification:**
- Grep all `.js` files in `assets/animation/` for layout-triggering properties in gsap.to/gsap.fromTo/gsap.from calls
- Zero violations should remain
</task>

<task id="4">
**Verify and strengthen mobile/coarse-pointer guards**

Ensure all complex animation modules properly gate behind pointer detection:

1. **parallax.js** — Already has `isCoarsePointer` guard. Verify it's at the top of init().
2. **scroll-skew.js** — Already has `isCoarsePointer` guard. Verify.
3. **webgl-bg.js** — Must gate behind `isCoarsePointer`. Check if it exits early for coarse pointer. If not, add gate.
4. **cursor.js** — Must exit if coarse pointer or no hover. Verify.
5. **magnetic.js** — Must exit if coarse pointer or no hover. Verify.
6. **card-effects.js** — Tilt must exit on coarse (already does). Scroll reveals should still work on mobile.
7. **hover-effects.js** — Complex hover should exit on coarse pointer.

For webgl-bg.js specifically:
- If `isCoarsePointer` is true, skip WebGL initialization entirely
- Show CSS gradient fallback instead (the context-loss fallback should already handle this pattern)

**Verification:**
- Each module listed above has an early-return guard checking `window.RR.breakpoints.isCoarsePointer`
- On a simulated touch device, WebGL canvas should not initialize
- Scroll-driven animations (scroll-reveals, section-choreography) should still work on mobile — only parallax, skew, WebGL, cursor, and magnetic are disabled
</task>
